Motivation: The Problem

Consider e-voting without a trusted authority (e.g., on a blockchain).

Naive Approach: Commit-Reveal
  • Voters commit to their vote (Phase 1)
  • Voters reveal their opening (Phase 2)
  • What if voters go offline before revealing?
  • Attacker can block openings to bias the result
Solution: Time-Lock Puzzles
  • Replace commitments with time-lock puzzles
  • Offline voters? Just solve their puzzles
  • But: 100,000 missing votes = 100,000 puzzles to solve!
  • Does not scale with many voters

Concept 1

Time-Lock Puzzles

Encrypting messages for the future

Time-Lock Puzzles (TLP)

Encapsulate a secret s such that it remains hidden until time T has elapsed.

Key Properties

  • Fast generation: Creating a puzzle is much faster than T
  • Sequential work: Secret hidden from any circuit of depth < T, regardless of parallelism
  • No shortcut: Throwing more processors at it does not help

RSA-based TLP (Rivest, Shamir, Wagner 1996) [1]

N = p · q   (RSA modulus)
Puzzle: (N, T, x, x2T · k, Enc(k, s))

Solving requires T sequential squarings:
x → x² → x4 → x8 → ... → x2T

Why Can't We Parallelize?

Repeated squaring modulo N is inherently sequential.

To compute x2T mod N, each squaring depends on the previous result:

w0 = x
→
w1 = w0²
→
w2 = w1²
→
...
→
wT = wT-1²

Without knowing p, q

Must perform all T squarings one after another.

1000 CPUs ≠ 1000x speedup

With knowing p, q (setup authority)

Can reduce exponent: 2T mod φ(N)

This allows fast puzzle generation!

x = 2, N = 21 (= 3 × 7), T = 3 squarings

Compute step by step:   2 → 2² = ? → ?² = ? → ?² mod 21 = ?

2 → 4 → 16 → 16² = 256 mod 21 = 4
Shortcut: φ(21) = 12,   2³ mod 12 = 8,   28 mod 21 = 256 mod 21 = 4 ✓   (same answer, one step!)

The Scalability Problem

Standard TLPs require solving each puzzle individually.

E-voting with n voters, k go offline

Puzzle 1
T squarings
Puzzle 2
T squarings
Puzzle 3
T squarings
...
Puzzle k
T squarings

Total work: k × T sequential squarings → potentially millions of puzzles!

The key insight of Malavolta & Thyagarajan

What if we could evaluate a function over puzzles first, then solve just one puzzle?

Combine all puzzles
instant
→
Solve 1 puzzle
T squarings

Complexity: O(n · T) → O(m · T) → for 1M voters (n) and 5 candidates (m), that's 5 puzzles instead of 1,000,000

Concept 2

Homomorphic Encryption

Computing on encrypted data without decrypting it

What is Homomorphic Encryption?

A cryptosystem where operations on ciphertexts correspond to operations on plaintexts.

Enc(a) ⊕ Enc(b)  =  Enc(a + b)

Without Homomorphism

Decrypt all
→
Compute f(x)
→
Re-encrypt

Requires access to the secret key → breaks privacy

With Homomorphism

Compute on ciphertexts directly
→
Decrypt once

No one needs to see the plaintext until the final result

Paillier addition with N = 7, N² = 49

Enc(s) = (1+N)s mod N²   →   Enc(2) = 8² mod 49 = 15    Enc(3) = 8³ mod 49 = 22

What is 15 × 22 mod 49? And what secret does it decode to?

15 × 22 = 330   →   330 mod 49 = 36   →   Decode: (36 − 1) / 7 = 5 = 2 + 3 ✓

Flavors of Homomorphism

Linearly Homomorphic

Supports addition of secrets.

Enc(a) · Enc(b)
= Enc(a + b)

Used for e-voting & coin flipping.

Efficient RSA-based construction.

Multiplicatively Homomorphic

Supports multiplication of secrets.

Enc(a) · Enc(b)
= Enc(a · b)

Used for contract signing.

Efficient RSA-based construction.

Fully Homomorphic [3]

Supports any function (NAND gates).

Eval(C, Enc(a), Enc(b))
= Enc(C(a, b))

Needed for sealed-bid auctions.

Requires obfuscation → theoretical [2].

Core Contribution

Homomorphic Time-Lock Puzzles

Eliminating the trusted key holder - decryption through time, not authority

HTLP: Four Algorithms

An HTLP scheme is defined by four algorithms.

Algorithm Description Key Feature
HP.PSetup(1λ, T) Generate public parameters pp One-time setup, fixes N, g, h
HP.PGen(pp, s) Create puzzle Z for secret s Fast (knows group order)
HP.PEval(C, pp, Z1..Zn) Evaluate C → outputs a new puzzle Creates new puzzle, no solving!
HP.PSolve(pp, Z) Solve a puzzle to recover secret T sequential squarings

PEval outputs a fresh puzzle encoding C(s1,...,sn) — it can be solved like any other puzzle, or fed into further PEval calls.

Compactness: the evaluated puzzle size and solving time are independent of the circuit size.

HTLP vs. Standard Homomorphic Encryption

Both compute on encrypted data — they differ in how decryption happens.

Standard HE

Single Point of Failure

  • Requires a trusted party holding the secret key
  • Key compromised → all encrypted data exposed
  • Counting is private, only the key holder can finalize
HTLP

No Trusted Key Holder

  • Anyone can solve the aggregated puzzle
  • No secret key, time itself decrypts
  • Counting is public and verifiable

Why this matters for blockchains

No trusted authority means the protocol is deployable on a public ledger. Anyone can run PEval to combine puzzles and PSolve to recover the result.

LHTLP — How It Works

Based on a Paillier-style encoding in Z*N². Four steps, no trusted party.

Step 1 · Setup

Generate the public parameters once.

N = p · q   (strong RSA)
g — generator of JN
h = g2T mod N
pp = (T, N, g, h)
Step 2 · Generate Puzzle

Encrypt secret s with random r.

u = gr mod N
v = hr·N · (1+N)s mod N²
Z = (u, v)
Step 3 · Combine (PEval)

Multiply puzzles component-wise. No solving needed.

ũ = ∏ ui mod N
ṽ = ∏ vi mod N²

→ Result encodes ∑ si

Step 4 · Solve (PSolve)

T sequential squarings, then decode.

w = u2T mod N
s = decode(v, wN)

Why Adding Puzzles Works

Each puzzle has two parts. They combine independently when puzzles are multiplied.

Secret Encoding
(1+N)s

Carries the secret s.

(1+N)a · (1+N)b = (1+N)a+b

→ Multiplying = adding the secrets

Time-Lock Mask
hr·N

Hides the encoding until time T.

hr1·N · hr2·N = h(r1+r2)·N

→ Masks combine in their own subgroup

Voting example: N = 7, 3 voters, 2 vote Codex (s = 1)

v₁ = (1+7)¹ = 8,   v₂ = 8,   v₃ = (1+7)⁰ = 1  →  Product mod 49 = ?

8 × 8 × 1 = 64 mod 49 = 15  →  Decode: (15 − 1) / 7 = 2 votes for Codex ✓

Multiplicatively Homomorphic TLP (MHTLP)

Same blueprint, different encoding. Swap Paillier for a Diffie-Hellman structure in JN.

LHTLP encoding
v = hr·N · (1+N)s mod N²

Combine: v1 · v2  encodes  s1 + s2

→ Used for sums (e.g. vote counts)

MHTLP encoding
v = hr · s mod N

Combine: v1 · v2  encodes  s1 · s2

→ Used for products (e.g. contract signing)

Useful Variant: XOR via Multiplication

Encode a bit s ∈ {0, 1} as (-1)s → multiplying encodings yields the XOR of the bits.

Security: Sequential Squaring + DDH over JN.

Multi-Party Coin Flipping

Fair randomness without trust

Multi-Party Coin Flipping

Goal: n parties flip a fair coin. Even n-1 colluders cannot bias the outcome.

Protocol (using LHTLP)

Setup
Generate pp
→
Each Party
Choose random bi
Zi ← PGen(pp, bi)
Broadcast Zi
→
Combine
Z ← PEval(⊕, Z1,...,Zn)
→
Result
s ← PSolve(Z)
coin = LSB(s)

Why it works

  • Each bit bi is hidden inside a time-lock puzzle
  • Even if n-1 parties collude, the last bit is unknown
  • Only one puzzle to solve (regardless of who goes offline)

E-Voting

Scalable, unbiased elections without a trusted authority

E-Voting Protocol

Setting: n voters, m candidates, no trusted tallying authority.

Voting Phase

Each voter casts an m-puzzle ballot

Encrypt 1 for the chosen candidate, 0 for everyone else.

Vote for j
→
PGen(0), …, PGen(1) at j, …, PGen(0)
→
Broadcast ballot

Example (2 candidates): Vote Codex = (PGen(1), PGen(0))    Vote Claude = (PGen(0), PGen(1))

Counting Phase

Combine first, solve once per candidate

All n ballots
→
PEval — homomorphic sum
per candidate
→
PSolve — recover total
T squarings

Solve m puzzles total, not n. For 1M voters and 5 candidates: 5 puzzles instead of up to 1M.

E-Voting Security Properties

🔒

Ballot Secrecy

Individual votes are hidden inside time-lock puzzles. The LHTLP security guarantees no one can decrypt before time T.

⚖

No Bias

Votes are committed before counting starts. The time-lock prevents anyone from peeking at partial results during the voting phase.

🛡

DoS Resilient

If voters go offline, their votes can still be counted via puzzle solving. No need for the voter's cooperation in the counting phase.

No Trusted Authority Needed

The counting operation is public, anyone can homomorphically combine puzzles and solve them. This makes the protocol suitable for blockchain-based deployment where the bulletin board is append-only and publicly readable.

Beyond LHTLP: Limitations & Follow-up Work

Limitations

More Complex Voting Needs More than LHTLP

  • LHTLP only supports addition: ideal for simple plurality voting
  • Ranked-choice: requires comparisons → MHTLP or fully homomorphic
  • Single Transferable Vote: needs richer arithmetic and tie-breaking logic
  • Verifiability: often paired with zero-knowledge proofs of well-formed votes

→ Real-world elections demand more sophisticated cryptographic stacks.

Follow-up Work (TCC '19)

Lattice-Based Time-Lock Puzzles [4]

  • Post-quantum security: survives a future quantum adversary
  • No RSA / factoring assumption: based on Learning With Errors (LWE)
  • Homomorphic operations over time-locked data with richer expressiveness
  • Trade-off: larger puzzle sizes and more complex setup than RSA-based HTLP

→ A promising direction for quantum-resistant time-lock cryptography.

Summary & Key Takeaways

⚡

1. Evaluate first, solve once

HTLPs reduce time-lock workload from O(n · T) to O(m · T), independent of the number of participants.

⛓

2. No trusted authority

Counting is public, anyone can run PEval and PSolve. A natural fit for blockchains and public bulletin boards.

🔧

3. Two practical flavors

LHTLP for additions (e-voting, coin flipping). MHTLP for multiplications (contract signing). Both efficient under standard RSA assumptions.

🔬

4. Open frontier

Practical fully-homomorphic TLP, public-coin setup, and post-quantum lattice-based HTLPs remain active research directions.

Your Questions and Our Answers

Q1. How is the time parameter T chosen, and who maintains it as hardware gets faster?

A. T = desired delay × fastest known squarings/sec, with a 2–4× safety margin for ASIC (application-specific integrated circuit) speed-ups. Open communities (drand, IACR) republish reference benchmarks over time.

Q2. What happens if a voter submits a puzzle that encodes garbage instead of a valid vote?

A. The tally would be corrupted. Defense: a NIZK (non-interactive zero-knowledge) proof of well-formedness, checked once at submission, a few KB, ≈ 5–50 ms per ballot.

Q3. Can the trusted setup be eliminated while keeping efficiency?

A. Open problem. Class groups (Wesolowski) give public-coin parameters at 5–10× overhead; a fully trustless RSA-style setup remains a research frontier.

Q4. How practical are HTLPs at million-voter scale?

A. Per ballot ≈ m × 768 B at 128-bit security. The win: solving cost is O(m·T) regardless of n, ballot storage moves to a data-availability layer.

Q5. Can fully homomorphic HTLP be built without indistinguishability obfuscation?

A. Partially. Brakerski–Döttling–Garg–Malavolta (TCC '19) [4] obtain homomorphic TLPs from rate-1 FHE (fully homomorphic encryption) under LWE, sidestepping iO (indistinguishability obfuscation). Fully compact FHTLP without iO and lattice-based HTLPs remain the active frontier.
