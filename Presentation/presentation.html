The Problem: Voting Without Trust

Imagine an election on a blockchain — no trusted authority to count votes.

Approach 1: Commit & Reveal
  • Voters commit to their vote, then reveal later
  • Problem: voters can go offline and never reveal
  • An attacker can block reveals to manipulate the outcome
Approach 2: Time-Lock Puzzles
  • Lock each vote in a puzzle that can be "brute-forced" after time T
  • Offline voters? Just solve their puzzle
  • But: 100k missing votes = 100k puzzles to solve!

Malavolta & Thyagarajan, Section 1.1

Concept 1

Homomorphic Encryption

Computing on encrypted data — without ever decrypting

Homomorphic Encryption in a Nutshell

You can do math on locked boxes without opening them.

Encrypt(3) + Encrypt(5)  =  Encrypt(8)

Normal Encryption

To add two secrets, you must decrypt first, compute, then re-encrypt.

Whoever computes sees the data!

Homomorphic Encryption

Combine the encrypted values directly. Decrypt only the final result.

Nobody sees individual values!

Analogy: Think of a ballot box. You drop sealed envelopes in. A magic box shakes them and outputs the total — without ever opening any envelope.

Types of Homomorphism

Additive (Linear)

Can add encrypted values.

Enc(a) · Enc(b)
= Enc(a + b)

Used for voting & coin flipping

Multiplicative

Can multiply encrypted values.

Enc(a) · Enc(b)
= Enc(a · b)

Used for contract signing

Fully Homomorphic

Can compute any function.

Eval(f, Enc(a), Enc(b))
= Enc(f(a,b))

Theoretical / not yet practical

This paper provides practical RSA-based constructions for additive & multiplicative (Sections 4.1, 4.2)

Concept 2

Time-Lock Puzzles

Encrypting a message for the future

Time-Lock Puzzles: The Idea

Lock a secret so that it cannot be opened until a certain amount of computation time has passed.

Two Key Properties

  • Fast to create: The person locking the puzzle does it in milliseconds (they know a shortcut)
  • Slow to open: Anyone else must do T sequential steps — one after the other, no shortcuts

How? Repeated Squaring (Rivest, Shamir, Wagner 1996)

Pick a number x, then square it T times modulo N (an RSA number):

x
→
x²
→
x4
→
x8
→
...
→
x2T

Each step depends on the previous one → cannot be parallelized (even with 1000 CPUs).

Creator vs. Solver

Why can the creator be fast but the solver is slow?

🔓

Puzzle Creator (knows p, q)

Knows the factorization of N = p · q

Can compute a shortcut:

e = 2T mod φ(N)
then: xe mod N

Done in milliseconds!

🔒

Puzzle Solver (doesn't know p, q)

Doesn't know the factorization

Must square step by step:

w0 = x
wi+1 = wi² mod N
... repeat T times

Takes ~T seconds!

The Scalability Problem

With standard TLPs, you must solve each puzzle separately.

Standard Approach: Solve every missing vote

Puzzle 1
~2.5 min
Puzzle 2
~2.5 min
Puzzle 3
~2.5 min
...
Puzzle n
~2.5 min

100,000 voters offline = 100,000 × 2.5 min ≈ 173 days

The Key Idea: Combine first, solve once

Combine all puzzles
instant (just multiplication)
→
Solve 1 result puzzle
~2.5 min per candidate

Any number of voters → only solve m puzzles (m = number of candidates)

Core Contribution

Homomorphic Time-Lock Puzzles

Time-Lock + Homomorphism = scalable crypto protocols

HTLP: Four Building Blocks

An HTLP extends time-lock puzzles with a homomorphic evaluation step.

Step What it does Speed
PSetup One-time setup: generate keys N, g, h Once (fast, knows factorization)
PGen Create a puzzle that locks a secret value Fast (milliseconds)
PEval Combine puzzles → result puzzle (no solving!) Instant (just multiplication)
PSolve Open a puzzle by doing T squarings Slow (T seconds, sequential)

The magic: PEval lets you combine n puzzles into 1 — and PSolve only needs to run once on the combined result. The solving time is the same whether you had 10 or 10 million input puzzles.

How LHTLP Works (Linear / Additive)

Uses the Paillier trick: in the group Z*N², multiplication of ciphertexts = addition of secrets.

Creating a Puzzle for secret s

Pick random r u = gr mod N v = hr·N · (1+N)s mod N² Puzzle = (u, v)

The secret s is hidden inside v, masked by the blinding factor hr·N.

Combining (PEval) — just multiply!

ũ = u1 · u2 · ... · un mod N ṽ = v1 · v2 · ... · vn mod N²

Result puzzle encodes s1 + s2 + ... + sn

Solving (PSolve)

w = u2T mod N (T squarings) sum = decode(v, w)

Paper Section 4.1 — Security: Sequential Squaring + DDH + DCR assumptions

Why Multiplying Puzzles = Adding Secrets

The encoding (1+N)s has a special property:

(1+N)a · (1+N)b  =  (1+N)a+b   mod N²

Concrete Example with 3 Votes

Alice gets votes with s=1, Bob's position gets s=0:

Vote 1: Alice
s=1 → (1+N)1
×
Vote 2: Bob
s=0 → (1+N)0
×
Vote 3: Alice
s=1 → (1+N)1
=
Combined
(1+N)2 → Alice: 2

The blinding factors (hr·N) also multiply — but they cancel out during PSolve!

Application 1

Multi-Party Coin Flipping

Fair randomness among n parties — no one can cheat

Coin Flipping with HTLP

Goal: n parties agree on a random bit. Even n−1 cheaters can't bias it.

Step 1
Each party picks a
random bit bi
→
Step 2
Lock it: Zi = PGen(bi)
Broadcast Zi
→
Step 3
Combine all:
Z = PEval(Z1,...,Zn)
→
Step 4
Solve Z, take
last bit = coin

Why it's fair

Each bit is time-locked. Even if n−1 parties collude, they can't see the last party's bit before committing their own.

Why it scales

Without HTLP:n × T squarings
With HTLP:1 × T squarings

1000 parties? Still just 1 puzzle to solve.

Paper Section 6.2 — Relevant for blockchain randomness (e.g., Ethereum)

Application 2

E-Voting

Scalable elections without a trusted authority

E-Voting with HTLP

n voters, m candidates. Each vote = a vector of m puzzles.

How a Voter Encodes Their Choice

Voting for candidate j? Create m puzzles:

Puzzle 1: PGen(0) ← not my pick Puzzle 2: PGen(1) ← my candidate! Puzzle 3: PGen(0) ← not my pick ...

In our demo (2 candidates):

Vote Alice:(PGen(1), PGen(0))
Vote Bob:(PGen(0), PGen(1))

Counting Phase

For each candidate j:

1. Combine all j-th puzzles
PEval — instant, just multiply
↓
2. Solve the combined puzzle
PSolve — T squarings
↓
Result: total votes for candidate j

Only m puzzles to solve — not n!

Why This Voting Scheme is Secure

🔒

Ballot Secrecy

Individual votes are locked in time-lock puzzles. No one can decrypt them before time T.

⚖

No Bias

All votes are committed before anyone can see partial results. The time-lock prevents peeking.

🛡

Offline-Resilient

Voters going offline is fine — their puzzles are already combined. No cooperation needed to count.

No trusted authority: The counting is a public operation. Anyone can combine puzzles and solve them. Perfect for blockchain deployment with a public append-only bulletin board.

Key Code Snippets

Puzzle Generation (PGen)

def PGen(self, s): r = random.randrange(1, self.N2) u = pow(self.g, r, self.N) v = (pow(self.h, r*self.N, self.N2) * pow(1+self.N, s, self.N2)) % self.N2 return (u, v)

Homomorphic Combine (PEval)

def PEval(self, puzzles): u_tilde, v_tilde = 1, 1 for u, v in puzzles: u_tilde = (u_tilde * u) % self.N v_tilde = (v_tilde * v) % self.N2 return (u_tilde, v_tilde)

Puzzle Solving (PSolve)

def PSolve(self, puzzle): u, v = puzzle w = u # T sequential squarings (~2.5 min) for i in range(self.T): w = pow(w, 2, self.N) # Decode the sum w_N_inv = mod_inverse( pow(w, self.N, self.N2), self.N2) return (((v*w_N_inv) % self.N2) - 1) // self.N

Casting a Vote

if candidate == 'Alice': vote = (PGen(1), PGen(0)) elif candidate == 'Bob': vote = (PGen(0), PGen(1))

Summary

What We Learned

  • Time-Lock Puzzles encrypt messages for the future via sequential squaring
  • Homomorphic Encryption allows computing on ciphertexts without decrypting
  • HTLP combines both: evaluate a function over puzzles, then solve just one

Paper Contributions

Scheme Operation Use Case
LHTLP Addition Voting, Coin Flip
MHTLP Multiplication Contract Signing

Extensions: reusable setup, combining different hardness parameters, semi-compact branching programs

Key Takeaways

💡

Evaluate first, solve once. HTLPs reduce computation from O(n · T) to O(m · T) — independent of the number of participants.

⛓

No trusted authority needed. Puzzle solving is public. Anyone can combine and verify. Ideal for blockchain-based protocols.

🔬

Open challenges: Efficient public-coin setup, practical fully-homomorphic TLP without obfuscation, and tighter security proofs.
