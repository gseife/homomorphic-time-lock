Motivation: The Problem

Consider e-voting without a trusted authority (e.g., on a blockchain).

Naive Approach: Commit-Reveal
  • Voters commit to their vote (Phase 1)
  • Voters reveal their opening (Phase 2)
  • What if voters go offline before revealing?
  • Attacker can block openings to bias the result
Solution: Time-Lock Puzzles
  • Replace commitments with time-lock puzzles
  • Offline voters? Just solve their puzzles
  • But: 100,000 missing votes = 100,000 puzzles to solve!
  • Does not scale with many voters

Malavolta & Thyagarajan, Section 1.1

Concept 1

Homomorphic Encryption

Computing on encrypted data without decrypting it

What is Homomorphic Encryption?

A cryptosystem where operations on ciphertexts correspond to operations on plaintexts.

Enc(a) ⊕ Enc(b)  =  Enc(a + b)

Without Homomorphism

Decrypt all
→
Compute f(x)
→
Re-encrypt

Requires access to the secret key — breaks privacy

With Homomorphism

Compute on ciphertexts directly
→
Decrypt once

No one needs to see the plaintext until the final result

Example (Paillier): Multiply two ciphertexts mod N² → get encryption of the sum of plaintexts. This is exactly the structure used in our HTLP construction!

Flavors of Homomorphism

Linearly Homomorphic

Supports addition of secrets.

Enc(a) · Enc(b)
= Enc(a + b)

Used for e-voting & coin flipping.

Efficient RSA-based construction.

Multiplicatively Homomorphic

Supports multiplication of secrets.

Enc(a) · Enc(b)
= Enc(a · b)

Used for contract signing.

Efficient RSA-based construction.

Fully Homomorphic

Supports any function (NAND gates).

Eval(C, Enc(a), Enc(b))
= Enc(C(a, b))

Needed for sealed-bid auctions.

Requires obfuscation — theoretical.

This paper focuses on practical RSA-based constructions: linear & multiplicative (Sections 4.1, 4.2)

Concept 2

Time-Lock Puzzles

Encrypting messages for the future

Time-Lock Puzzles (TLP)

Encapsulate a secret s such that it remains hidden until time T has elapsed.

Key Properties

  • Fast generation: Creating a puzzle is much faster than T
  • Sequential work: Secret hidden from any circuit of depth < T, regardless of parallelism
  • No shortcut: Throwing more processors at it does not help

RSA-based TLP (Rivest, Shamir, Wagner 1996)

N = p · q   (RSA modulus)
Puzzle: (N, T, x, x2T · k, Enc(k, s))

Solving requires T sequential squarings:
x → x² → x4 → x8 → ... → x2T

Rivest, Shamir, Wagner: "Time-lock puzzles and timed-release crypto" (1996)

Why Can't We Parallelize?

Repeated squaring modulo N is inherently sequential.

To compute x2T mod N, each squaring depends on the previous result:

w0 = x
→
w1 = w0²
→
w2 = w1²
→
...
→
wT = wT-1²

Without knowing p, q

Must perform all T squarings one after another.

1000 CPUs ≠ 1000x speedup

With knowing p, q (setup authority)

Can reduce exponent: 2T mod φ(N)

This allows fast puzzle generation!

Sequential Squaring Assumption (Assumption 1 in the paper)

The Scalability Problem

Standard TLPs require solving each puzzle individually.

E-voting with n voters, k go offline

Puzzle 1
T squarings
Puzzle 2
T squarings
Puzzle 3
T squarings
...
Puzzle k
T squarings

Total work: k × T sequential squarings — potentially millions of puzzles!

The key insight of Malavolta & Thyagarajan

What if we could evaluate a function over puzzles first, then solve just one puzzle?

Combine all puzzles
instant
→
Solve 1 puzzle
T squarings

Total work: m × T (m = number of candidates, not voters!)

Core Contribution

Homomorphic Time-Lock Puzzles

Combining the best of both worlds

HTLP: Formal Definition

Four algorithms define an HTLP scheme:

Algorithm Description Key Feature
HP.PSetup(1λ, T) Generate public parameters pp One-time setup, fixes N, g, h
HP.PGen(pp, s) Create puzzle Z for secret s Fast (knows group order)
HP.PEval(C, pp, Z1..Zn) Evaluate function C over puzzles No solving needed!
HP.PSolve(pp, Z) Solve a puzzle to recover secret T sequential squarings

Compactness: The size of the evaluated puzzle and solving time are independent of the circuit size. This is what makes HTLP useful — otherwise, just solving all puzzles individually would suffice.

LHTLP Construction (Linearly Homomorphic)

Based on Paillier-style encoding in Z*N².

Setup

N = p · q   (strong RSA integer) g := -g̃² mod N   (generator of JN) h := g2T mod N   (using φ(N)) pp = (T, N, g, h)

Puzzle Generation (PGen)

r ← {1, ..., N²} u := gr mod N v := hr·N · (1+N)s mod N² Z := (u, v)

Homomorphic Evaluation (PEval)

// Just multiply component-wise! ũ := ∏ ui mod N ṽ := ∏ vi mod N² // Result: puzzle encoding ∑ si

Solving (PSolve)

// T sequential squarings w := u2T mod N // Recover sum via Paillier decoding s := (v · (wN)-1 mod N² - 1) / N

Paper Section 4.1 — Security: Sequential Squaring + DDH over JN + DCR over Z*N²

Why Homomorphic Addition Works

The algebraic structure of Z*N² enables "addition through multiplication".

Key Insight: The Paillier Structure

The group Z*N² decomposes into two independent components:

Z*N²  ≅  ⟨1+N⟩  ×  {xN : x ∈ Z*N}
  • (1+N)s encodes the secret s — and (1+N)a · (1+N)b = (1+N)a+b
  • hr·N is the "blinding factor" (one-time pad) — it lives in a separate subgroup

Multiplying Two Puzzles

v1 · v2 = hr1·N(1+N)s1 · hr2·N(1+N)s2
         = h(r1+r2)·N · (1+N)s1+s2 mod N²

The blinding factors combine, and the secrets add up. Solving recovers s1 + s2.

Multiplicatively Homomorphic TLP (MHTLP)

Switching the encoding from Paillier to Diffie-Hellman structure in JN.

Construction

// Setup: same as LHTLP pp = (T, N, g, h = g2T) // PGen: secret s ∈ JN r ← {1, ..., N²} u := gr mod N v := hr · s mod N  // DH-style // PSolve w := u2T mod N s := v / w mod N

Homomorphic Multiplication

v1 · v2 = hr1·s1 · hr2·s2
         = hr1+r2 · (s1 · s2) mod N

Secrets are multiplied when puzzles are combined.

XOR variant: Encode s ∈ {0,1} as (-1)s → multiplication becomes XOR.

Security: Sequential Squaring + DDH over JN

Paper Section 4.2

Application 1

Multi-Party Coin Flipping

Fair randomness without trust

Multi-Party Coin Flipping

Goal: n parties flip a fair coin. Even n-1 colluders cannot bias the outcome.

Protocol (using LHTLP)

Setup
Generate pp
→
Each Party
Choose random bi
Zi ← PGen(pp, bi)
Broadcast Zi
→
Combine
Z ← PEval(⊕, Z1,...,Zn)
→
Result
s ← PSolve(Z)
coin = LSB(s)

Why it works

  • Each bit bi is hidden inside a time-lock puzzle
  • Even if n-1 parties collude, the last bit is unknown
  • Only one puzzle to solve (regardless of who goes offline)

Complexity Comparison

Classical TLPn × T squarings
HTLP1 × T squarings

Application 2

E-Voting

Scalable, unbiased elections without a trusted authority

E-Voting Protocol

Setting: n voters, m candidates. No trusted tallying authority.

Voting Phase

Each voter Vi voting for candidate j creates:

votei = (Z1, ..., Zm) where Zk = PGen(pp, 0) for k ≠ j Zj = PGen(pp, 1) for chosen candidate

Example with 2 candidates (Alice, Bob):

Vote Alice:(PGen(1), PGen(0))
Vote Bob:(PGen(0), PGen(1))

Counting Phase

For each candidate j:

// 1. Homomorphically sum all votes Z̃j = PEval(⊕, Zj(1), ..., Zj(n)) // 2. Solve ONE puzzle per candidate vj = PSolve(pp, Z̃j) // vj = total votes for candidate j

Solve m puzzles total — not n!

For 1M voters, 5 candidates: 5 puzzles instead of up to 1M

Paper Section 6.1

E-Voting Security Properties

🔒

Ballot Secrecy

Individual votes are hidden inside time-lock puzzles. The LHTLP security guarantees no one can decrypt before time T.

⚖

No Bias

Votes are committed before counting starts. The time-lock prevents anyone from peeking at partial results during the voting phase.

🛡

DoS Resilient

If voters go offline, their votes can still be counted via puzzle solving. No need for the voter's cooperation in the counting phase.

No Trusted Authority Needed

The counting operation is public — anyone can homomorphically combine puzzles and solve them. This makes the protocol suitable for blockchain-based deployment where the bulletin board is append-only and publicly readable.

Code: The LHTLP Class

Puzzle Generation

def PGen(self, s): r = random.randrange(1, self.N2) u = pow(self.g, r, self.N) v = (pow(self.h, r * self.N, self.N2) * pow(1 + self.N, s, self.N2) ) % self.N2 return (u, v)

Homomorphic Evaluation

def PEval(self, puzzles): u_tilde, v_tilde = 1, 1 for u, v in puzzles: u_tilde = (u_tilde * u) % self.N v_tilde = (v_tilde * v) % self.N2 return (u_tilde, v_tilde)

Just multiplication — runs in microseconds regardless of vote count.

Puzzle Solving (The Bottleneck)

def PSolve(self, puzzle): u, v = puzzle w = u # T sequential squarings for i in range(self.T): w = pow(w, 2, self.N) # Paillier-style decoding w_N_inv = mod_inverse( pow(w, self.N, self.N2), self.N2) return (((v * w_N_inv) % self.N2) - 1) // self.N

Vote Casting

if candidate == 'Alice': vote = (PGen(1), PGen(0)) elif candidate == 'Bob': vote = (PGen(0), PGen(1)) votes.append(vote)

Summary

What We Covered

  • Time-Lock Puzzles: encrypt messages for the future using sequential squaring
  • Homomorphism: compute on ciphertexts without decrypting
  • HTLP: combines both — evaluate functions over puzzles, then solve once
  • Coin Flipping: n parties, 1 puzzle to solve
  • E-Voting: n voters, m puzzles to solve (not n!)
  • Live Demo: real LHTLP voting running on Flask

Constructions in the Paper

Scheme Operation Assumption
LHTLP Addition Seq. Sq. + DDH + DCR
MHTLP Multiplication Seq. Sq. + DDH

Extensions

  • Reusable setup (Thm. 4 & 5)
  • Semi-compact scheme for branching programs
  • Combining puzzles with different T values
  • Public-coin setup (open problem)

Key Takeaways

💡

Core Innovation: Evaluate first, solve once. HTLPs reduce the computational burden of time-lock-based protocols from O(n · T) to O(m · T) — independent of the number of participants.

⛓

Blockchain-Ready: No trusted authority needed. Puzzles are posted on a public bulletin board. Anyone can perform the counting — perfect for decentralized applications.

🔬

Open Questions: Efficient public-coin setup, practical fully-homomorphic TLP without obfuscation, and tighter security reductions remain fascinating challenges.
